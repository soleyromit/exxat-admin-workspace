1. Executive Summary
Post Course evaluation (PCE) is a non-discretionary accreditation requirement for every health profession program Exxat serves. The demand already exists within the installed base — yet clients are currently fulfilling it through third-party platforms, leaving a meaningful and recoverable gap in per-account revenue. This document does not make the case for whether to build; that decision has been made. It addresses how to build with intention — defining what is required to meet market expectations, what will differentiate Prism from established incumbents, and what will make it the platform clients choose to stay on.

This is a master PRD that covers following requirements:
1. Create Template
2. Push Survey end to end

2. Problem Statement
Health professions programs are obligated to run Course evaluations every semester, yet the tools currently available to them create more administrative burden than they resolve. Four distinct problems characterize the current state.

Key Product Decisions for Phase 1:
1. Scope is at the program level; not at tenant-level.
2. Semester (Term + Academic Year), start date, end date and Course offerings (courseID, Course Name, Term, Academic Year) must be created in PRISM to enable CE distribution.
3. For Clinical Course, we will keep preceptor and site evaluations within learning activities, rather than CE.
4. The overall approach is to re-design Surveys in such a way that Course Evaluation and Survey can co-exist.
5. Introduce 3 things: a. Survey type (mandatory) i. Course evaluations ii. General surveys - (Programmatic Surveys) b. Course type (Optional) i. Didactic ii. Clinical c. Subject (assigned to each section)
6. What will not be covered in phase 1: Detailed scope here

3. User Personas
Persona Cards
Faculty (Evaluated persona)
1. Instructor: Role: Designs and teaches didactic courses. Owns course objectives, syllabus, and mapping to accreditation standards for their course. Primary need: Specific, actionable course improvement feedback.
2. Course Coordinator: Role: responsible for course syllabus, objectives, sequence of content delivery, assessment planning, communication with students, reviewing struggling students and curriculum mapping to stds. Primary need for CE: Need to view the feedback on course material and own self.
Admin (Primary Operator): Role: Day-to-day admin of the program. Build survey templates. Sends, tracks, and follows up on all evaluations. Pulls reports. Primary need: Full automation of survey sends, reminder, and escalation workflows.
Student (Primary Respondent): Role: Enrolled in classroom or online learning. Fills out course and faculty evaluations each semester.

Limitation of Current Survey Capability (Exxat Surveys)
1. Per course distribution only, no bulk push available
2. Auto-render sections based on the count of subjects
3. Access control for analytics – within the form

4. Key User Flows
Flow Overview — The Full Lifecycle
Step 1: Template Creation → Admin
Step 2: Distribute Survey → Admin (Setup + Scheduling + distributing)
Step 3: Response Collection → Student
Step 4: Review and Enable Results → Admin
Step 5: Analytics → Faculty, Admins

Status of a Survey:
Status Label | Description
Draft | Same as current
Scheduled | The survey is fully configured and deployed. It will automatically go live on the defined start date and time.
Live | The survey is open and accepting responses
Closed, Pending Review | The survey window has closed. Responses are collected, but results are held, awaiting admin or coordinator for review before being released.
Closed, Results Available | The review is complete and results have been released. Authorized roles (Faculty, Program Directors, Admins) can now access reports and analytics.
Archived | <ignore for now>

Step 1 — Template Creation
Persona: Admin
Entry: Templates → "New Template"
- User gives the template a name and optional description
- User selects Course type: Didactic or Clinical (optional)
- User adds Section, adds section title, assigns a subject (List here) to each section
  - Adds Question
  - Gives answer type
- User can reorder section or questions via drag
- User saves the template

Step 2 — Create Survey
Persona: Admin
Entry: Survey Home → "Set Up Surveys" (wizard)
"End to End" Distribute Survey will have followings steps:
1. Properties/Details
2. Scope/Courses
3. Survey Design
4. Communication (newly added)
5. Report Access (newly added)
- User selects Term, Academic Year, and Program to scope the survey batch
- System loads all course offerings for that scope; all are selected by default
- User deselects any courses to exclude from the batch
- User assigns an approved template to each course (one at a time). If system only has one template created, all courses will auto assign that template.
- Bulk-assign option applies one template to all courses of the same type at once
- User reviews each course's faculty list and enrolled students; can add or remove from a slide-in drawer (Faculty tab, Students tab, Template tab)
- User sets survey open and close dates; bulk date override applies one date range to all courses
- System validates: close date must be after open date; all included courses must have a template
- User reviews a final summary (course count, student count, date range) and clicks "Push Surveys"
- System creates one Survey Configuration per course with status Scheduled (future open date) or Live (open date today or past)
- On success: Dashboard showing pushed course count and links to view surveys

Step 3 — Response Collection
Persona: Student
Entry: Email notification link or Exxat student portal → Surveys tab
- Student receives an email when a survey opens, with a direct link to the survey home (not links for each survey)
- Survey is also accessible from the student portal (Entry point TBD)
- Survey renders one section at a time (Subject: Course Content, Instructor, Course Coordinator)
- Each question shows the question text, response type (Likert scale, Yes/No, open text), and optional help text
- Required questions must be answered before the student can advance to the next section
- Student can navigate back to a previous section to change answers before final submission
- Student can also save the responses in draft mode
- Student submits the survey; submission is confirmed with a thank-you screen
- Once submitted, the survey moves to submitted state
- Student cannot see any results — not their own answers, not aggregate scores
- Survey is anonymous: no student identifier is attached to the submitted response

Step 4 — Review and Enable Results to Faculty
Persona: Admin
Entry: Survey Home → select a closed survey → "Moderate" tab
- Survey must be in "Closed but Pending Review" status to enter moderation
- Admin sees all open-text responses listed, one per card
- Each card shows the question text and the student's response; student identity is never shown
- Admin can flag a response as inappropriate (hides it from faculty view)
- Admin can unflag a previously flagged comment
- Flagged responses are excluded from faculty-facing results but remain in the admin audit trail
- Admin reviews all responses and clicks "Enable Results" to release results to faculty
- Once enabled, survey status changes to "Results Available"
- Admin cannot edit or delete response text — only flag/unflag
- If fewer than the minimum response threshold (e.g. 5) are received, the system warns the admin before enabling; admin can still proceed

9. Non-Functional Requirements
Category | Requirement
Anonymity / Privacy | Survey responses must be untraceable to individual students. No IP address, device ID, or response timing metadata exposed. FERPA compliant. We will just track submission and not the responses
Minimum N enforcement | Results for any course suppressed from all non-student views until minimum 5 responses received. Suppression rule visible to students as a trust signal before submission
Scoring standardisation | 1 – 5 Likert scale enforced as standard across all evaluations. Anchor labels standardised. Not a configurable option at faculty or program level
Security | Role-based access enforced at API level. Dean view cannot expose individual student or faculty data. All role transitions require re-authentication
Accessibility | WCAG 2.1 AA compliant across all student-facing survey interfaces. Screen reader compatible. Mobile-first responsive design
Data retention | Evaluation data retained for minimum 7 years to support longitudinal accreditation evidence (CAPTE requires 5-year continuous improvement documentation)
Open-text handling | Free-text responses stored encrypted. AI analysis (Phase 2) runs on anonymised text with no PII. Program director view of open-text shows per-course aggregate themes — not individual verbatims
