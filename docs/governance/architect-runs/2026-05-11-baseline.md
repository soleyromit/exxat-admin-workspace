### PROPOSED-CONSOLIDATE-1: `card-imposter-div` + `eyebrow-paragraph-outside-card` → `card-shape-masquerade`

- **Evidence**: both rules surface the same Card-shape-masquerade bug class with different facets.

- **Proposed unified rule**: `card-shape-masquerade` — single rule body that matches either (a) a `<div>` with `rounded` + `border` + `padding` without a `<Card` ancestor, OR (b) a top-level `<p>` with eyebrow styling (`uppercase`, `tracking-wider`, or `text-xs text-muted-foreground` as eyebrow) not wrapped in `<CardDescription>`.
- **Migration**: one-line edit to `pre-commit:120` (audit already runs both via `--strict`; consolidation is at the rule-definition level inside `scripts/ds-adoption-audit.py`). The `eyebrow-paragraph-outside-card` slug becomes an alias that resolves to `card-shape-masquerade` in output, so external citations still resolve.
- **Why this is the right move**: today both rules will fire on the same div if it has eyebrow children — false-positive noise. Consolidation aligns the user-facing message with the actual bug shape (Card chrome reinvention, regardless of which corner of it leaks).
- **Acknowledged tension**: `card-imposter-div` has a known false positive at `pce-modals.tsx:106` (the Checkbox-group wrapper) per `ds-escalations-2026-05-11.md:160` ("DS gap S4 — design decision needed"). See open question #2 below.
- **Decision (2026-05-11):** ACCEPTED. Both scan functions in `scripts/ds-adoption-audit.py` now emit `rule="card-shape-masquerade"` with facet-specific messages ("div facet" / "eyebrow facet"). Audit re-run: 0 hits for the consolidated rule.
