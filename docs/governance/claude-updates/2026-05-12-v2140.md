## ADOPT proposals (1 / 3 budget used)

### ADOPT-1: Upgrade `subagent_type` references from `general-purpose` workaround to readable display names

- **Upstream evidence**: Claude Code CHANGELOG 2.1.140 ‚Äî *"Improved Agent tool `subagent_type` matching to accept case- and separator-insensitive values (e.g. `"Code Reviewer"` resolves to `code-reviewer`)"* (`docs/governance/claude-updates/pending-review.md:19`).
- **Status upstream**: GA (main CHANGELOG body, no "Research Preview" tag).
- **What it gives us**: our two watcher slash commands currently include a defensive comment that forces `subagent_type: general-purpose` because custom subagent types were case/separator-sensitive and the runtime's matching was fragile for display-style names. That comment now reads as a permanent workaround when it's no longer needed. More importantly, when the parent agent spawns subagents like `ds-adoption-reviewer`, `verification-reviewer`, `visual-review`, `state-review`, or `architect`, it can now pass the display name naturally (e.g., `"DS Adoption Reviewer"` ‚Üí resolves to `ds-adoption-reviewer`) or use the slug directly without worrying about case. This eliminates one class of "subagent didn't fire because the type didn't match" bug that the commands comment was hedging against.
- **How to wire**: one-file change per command, plus a note update in SUBAGENTS.md if it documents the `general-purpose` workaround.
  1. `.claude/commands/check-claude-updates.md:16` ‚Äî replace:
     ```
     - subagent_type: `general-purpose` (until Claude Code resolves custom subagent types by filename ‚Äî confirm in `.claude/agents/claude-updates-watcher.md`)
     ```
     with:
     ```
     - subagent_type: `claude-updates-watcher` (Claude Code 2.1.140+ resolves agent names case- and separator-insensitively, so this now works directly)
     ```
  2. `.claude/commands/check-ds-updates.md:16` ‚Äî same pattern, change to `subagent_type: ds-updates-watcher`.
  3. *(Optional, low priority)* Check `docs/SUBAGENTS.md` for any guidance that tells the parent to use `general-purpose` as a fallback for custom agents ‚Äî update accordingly if found.
- **Replaces / consolidates with**: removes the `general-purpose` workaround caveat that has been in these two command files since they were authored. No file is deleted; the change is a 1-line edit per command file.
- **Risk**: very low. The change only affects the `subagent_type` value passed in the Agent tool call. If the local CLI build doesn't yet have 2.1.140 shipped, `claude-updates-watcher` would fail to resolve and the agent would fall through to a generic subagent ‚Äî same outcome as the old `general-purpose`. Rollback = revert both command files. Smoke test: invoke `/check-claude-updates` once after the edit and confirm the watcher subagent runs with its spec loaded (it'll report "`.claude/agents/claude-updates-watcher.md`" in its source line).

### `Read` tool `offset` validation fix

- **Upstream**: Claude Code 2.1.140 ‚Äî *"Fixed `Read` tool calls failing validation when `offset` is passed as a whitespace-padded or `+`-prefixed string."*
- **Ours**: `.claude/hooks/session-start.py`, `.claude/hooks/subagent-stop.py`, and `scripts/architect-input.py` all call `Read` (via the Claude Code runtime) but pass offset as a clean integer, not a whitespace-padded string. The fix silently benefits these call sites by making the runtime more lenient about the form.
- **Gap**: none ‚Äî we weren't hitting this bug. No action needed.
