### ADOPT-1: Add `claude-sonnet-5*` to model allow-list in settings.json

- **Upstream evidence**: Claude Code CHANGELOG 2.1.197 — *"Introducing Claude Sonnet 5: now the default model in Claude Code, with a native 1M-token context window and promotional pricing of $2/$10 per Mtok through August 31. Update to version 2.1.197 for access."*
- **Status upstream**: GA (main CHANGELOG body; 2.1.197 stable release; Anthropic news announcement cited).
- **What it gives us**: our `permissions.allow` model gate (`.claude/settings.json:15-16`) currently permits `Agent(model:claude-opus-4-8)` and `Agent(model:claude-sonnet-4-6)`. Any subagent spec that uses `claude-sonnet-5` (or any versioned variant like `claude-sonnet-5-20251120`) will hit a permission gate requiring interactive confirmation. As we upgrade subagent specs from claude-sonnet-4-6 to Sonnet 5, every spawn of those agents will block until Romit approves.

  The MEMORY.md rule "Design on Opus, implement-from-spec on Sonnet" still applies. The routing is unchanged: opus-4-8 for main design sessions (`.claude/settings.json:1`), sonnet-class for implementation subagents. Sonnet 5's 1M context window and quality improvements make it a better choice than sonnet-4-6 for our implementation subagents. Adding the wildcard entry proactively avoids permission friction when individual subagent specs are upgraded.

  Does Sonnet 5 change our Opus-vs-Sonnet routing rule? No. Opus 4.8 remains the main model for design judgment (the MEMORY.md rule is about capability tier, not a specific version). Sonnet 5 may be superior to Sonnet 4.6 but it's still a Sonnet-class model appropriate for implementation-from-spec work. Routing is unchanged; the allow-list entry just needs to expand.

- **How to wire**: one addition to `.claude/settings.json:16`, inside `permissions.allow`, after the existing `"Agent(model:claude-sonnet-4-6)"` entry:
  ```json
  "Agent(model:claude-sonnet-5*)"
  ```
  Use the wildcard `*` to cover both bare `claude-sonnet-5` and any date-versioned form (e.g., `claude-sonnet-5-20251120`), matching the existing `"Agent(model:claude-haiku*)"` pattern in the deny list.

  **Caveat**: Anthropic's exact model ID string for Sonnet 5 is not confirmed from the changelog text alone (the text says "Claude Sonnet 5" without the API identifier). Before applying, run `claude /model` in an interactive session and inspect the model list to confirm the exact ID. If the ID is not `claude-sonnet-5*`, update the pattern accordingly. Do NOT add without confirming the ID — a wildcard that's too broad could accidentally permit unintended models.

- **Replaces / consolidates with**: nothing removed. Additive to the existing allow list alongside `Agent(model:claude-sonnet-4-6)`, which remains for any subagent spec not yet upgraded.
- **Risk**: low. The wildcard `claude-sonnet-5*` is narrow — it only matches Sonnet 5 identifiers, not Claude 3 or other families. Rollback = remove the one line.

### ADOPT-2: Wire `Notification` hook for `agent_completed` → `subagent-stop.py`

- **Upstream evidence**: Claude Code CHANGELOG 2.1.198 — *"Added background agent notifications in `claude agents` — sessions that need input or finish now fire the `Notification` hook (`agent_needs_input` / `agent_completed`)."*
- **Status upstream**: GA (main CHANGELOG body; 2.1.198 stable release).
- **What it gives us**: our current `SubagentStop` hook (`.claude/settings.json:200-209`, wired to `subagent-stop.py`) fires when a subagent finishes within a session. But when an agent is launched as a background session via `claude agents` (which is how background subagents now run by default per CC 2.1.198), the `SubagentStop` event does not fire — instead the new `Notification` event fires with `agent_completed`. The result: background agent completions currently produce no hook-side logging, while in-session subagent completions do. Adding a `Notification` hook entry closes this gap and gives us a unified completion log regardless of whether the subagent ran in-session or as a background session.

  Concretely: our `subagent-stop.py` already reads the hook input payload and logs completion metadata to the subagent-invocations log. We can route the `Notification` event to the same script (it will receive a different payload format, so the script needs a small conditional to handle `agent_completed` events). This is one settings.json entry and a few lines in the existing script.

- **How to wire**: two changes.

  1. `.claude/settings.json` — add a `Notification` key inside the `hooks` object (after `SubagentStop`, before `Stop`):
     ```json
     "Notification": [
       {
         "matcher": "agent_completed",
         "hooks": [
           {
             "type": "command",
             "command": "python3 \"$CLAUDE_PROJECT_DIR\"/.claude/hooks/subagent-stop.py",
             "timeout": 5
           }
         ]
       }
     ]
     ```

  2. `.claude/hooks/subagent-stop.py` — add a branch at the top of the payload-reading logic to detect `Notification` events (the `hook_event_name` field will be `"Notification"` with a `data.reason` of `"agent_completed"`) and extract the relevant fields (session_id, agent name) from the Notification payload format rather than the SubagentStop format. The existing log-write path can be reused.

- **Replaces / consolidates with**: does not replace `SubagentStop`. The two hooks cover different execution contexts: SubagentStop (in-session subagents) and Notification/agent_completed (background sessions). Both route to the same logger.
- **Risk**: low. The `Notification` event matcher is narrow (`agent_completed` only — we do not wire `agent_needs_input`, which would fire for every interactive pause). If `subagent-stop.py` cannot parse the Notification payload format, it should fail gracefully (exit 0, log a parse warning) without blocking the agent. Rollback = remove the Notification block from settings.json.

### ADOPT-3: Log `prompt_id` from hook input payloads in `_telemetry.py`

- **Upstream evidence**: Agent SDK TypeScript CHANGELOG 0.3.196 — *"Added `prompt_id` field to hook input payloads for correlating hook events with OpenTelemetry prompt-level events."*
- **Status upstream**: GA (TS SDK 0.3.196; mirrors Claude Code runtime support added in the same cycle).
- **What it gives us**: every hook in our workspace (session-start.py, pre-tool-use.py, exxat-brief-gate.mjs, post-edit-checks.py, subagent-stop.py, etc.) receives a JSON payload on stdin. This payload now includes a `prompt_id` field. Our `_telemetry.py` shared helper (`.claude/hooks/_telemetry.py`) writes event records to the telemetry sink. Currently it does not capture `prompt_id`.

  Adding `prompt_id` to the `_telemetry.py` event record is the minimum-cost investment that makes our telemetry events correlatable to future OTEL pipelines. It does not require a running OTEL collector — it just enriches the log records so that IF an OTEL pipeline is stood up later (the `OTEL_LOG_TOOL_DETAILS=1` thread from v2154-2157 ADOPT-2), the correlation is already there. It also lets us manually grep the telemetry log to find all hook events that fired within a single prompt turn.

  This directly closes an open gap from `2026-05-12-baseline.md:54`: *"we'd need to know which tools each subagent called"* — `prompt_id` is the join key that ties tool events back to the originating prompt without requiring the full OTEL pipeline.

- **How to wire**: one-file change.
  - `.claude/hooks/_telemetry.py` — wherever the script reads the hook input payload from stdin (the `json.loads(sys.stdin.read())` call), extract `prompt_id` if present:
    ```python
    payload = json.loads(sys.stdin.read()) if not sys.stdin.isatty() else {}
    prompt_id = payload.get("prompt_id")   # new — may be None for older CLI builds
    ```
    Then include `"prompt_id": prompt_id` in the event dict passed to the log-write function. Callers that don't use `_telemetry.py` directly (e.g., `exxat-brief-gate.mjs`) are unaffected.
- **Replaces / consolidates with**: nothing removed. Purely additive field in existing event records.
- **Risk**: near-zero. `prompt_id` is extracted with `.get()` — absent on older CLI builds it will be `None`, which is logged as `null`. No behavior change; log records get one optional additional field. Rollback = remove the two added lines.

### Defect A: Fetch failure is indistinguishable from "no changes" (`scripts/claude-updates-watch.py:107-108`)

**Current behavior**: when `fetch()` returns `None` (network error, timeout, 404), the script executes `continue` at line 108, skipping that source. If ALL three sources fail, `changes = []`. The script then reaches the `if not changes:` branch (line 136) and writes `"no changes detected"` to `pending-review.md`. Result: a total fetch failure looks identical to a no-change run.

**Root cause in code**:
```python
# lines 106-108
text = fetch(src["url"])
if text is None:
    continue          # ← silent skip; changes remains empty
```

**Recommended fix**: track fetch failures and surface them in `pending-review.md`:
```python
fetch_failures = []
...
text = fetch(src["url"])
if text is None:
    fetch_failures.append(src_id)
```
