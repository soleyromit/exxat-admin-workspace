## ADOPT proposals (3 / 3 budget used)

### ADOPT-1: OTEL `agent_id` / `parent_agent_id` spans + per-subagent HTTP headers

- **Upstream evidence**: Claude Code CHANGELOG 2.1.139 — *"API requests from subagents now carry `x-claude-code-agent-id` / `x-claude-code-parent-agent-id` headers, and `claude_code.llm_request` OTEL spans include `agent_id` / `parent_agent_id` attributes."* (`docs/governance/claude-updates/pending-review.md:34`).
- **Status upstream**: GA (shipped in the main CHANGELOG body, no "Research Preview" tag).
- **What it gives us**: Native, structured per-subagent telemetry — without a hand-rolled wrapper script. Today `docs/governance/subagent-invocations.log` is empty despite the `SubagentStop` hook landing 2026-05-09 (see `docs/governance/architect-runs/2026-05-11-baseline.md:28`, "*the SubagentStop hook landed 2026-05-09 (commit `f9512037`) but isn't appending*"). The architect's open question #3 (`architect-runs/2026-05-11-baseline.md:149-152`) explicitly asks whether to build that wrapper. The wrapper IS NOW UPSTREAM as OTEL-native — every LLM request a subagent makes is already tagged at the protocol layer, so we get per-subagent fan-out, duration distributions, and tool-call counts for free if we point an OTEL collector at it. `claude_code.active_time.total` metric is also re-emitted in `--print` mode per the 2.1.139 bug-fix list (same changelog).
- **How to wire**: two-file change.
  1. `.claude/settings.json` — add a top-level `"env"` block (or `"telemetry"` block per upstream config schema; the watcher should confirm exact key names by fetching `https://code.claude.com/docs/en/monitoring-usage#opentelemetry` when applying) setting `CLAUDE_CODE_ENABLE_TELEMETRY=1` and `OTEL_EXPORTER_OTLP_ENDPOINT=` to a local collector (recommended: a file-exporter pointed at `docs/telemetry/otel-spans.jsonl`).
  2. `.claude/hooks/subagent-stop.py:50-55` — replace the `_telemetry_emit("subagent.stop", agent_type=..., ...)` call with a no-op IF the OTEL pipeline is verified live; otherwise keep as belt-and-braces. The `_telemetry.py` helper at `.claude/hooks/_telemetry.py:12` writes `docs/telemetry/events.jsonl` — keep that file for hook-emitted events, but stop pretending it's the subagent-invocation log.
- **Replaces / consolidates with**: `docs/governance/subagent-invocations.log` (currently 4-line header with zero data rows) becomes obsolete; `docs/governance/architect-runs/2026-05-11-baseline.md:149-152` open Q #3 closes with verdict "use upstream OTEL, don't build the wrapper". Architect's *next* run reads OTEL spans (or a digest of them) instead of waiting on the log.
- **Risk**: low-medium. Risks: (a) OTEL collector setup adds a moving part we now own — mitigate by starting with a file-exporter (zero infra). (b) If the OTEL config key names differ from what I've inferred, the architect won't get data until corrected — mitigate by Romit verifying with `https://code.claude.com/docs/en/monitoring-usage` before flipping. (c) Telemetry export may include tokens / prompt content depending on collector config — start with a metrics-only collector, not span-body export.

### ADOPT-2: Hook `args: string[]` exec form

- **Upstream evidence**: Claude Code CHANGELOG 2.1.139 — *"Added hook `args: string[]` field (exec form) that spawns the command directly without a shell, so path placeholders never need quoting."* (`docs/governance/claude-updates/pending-review.md:24`).
- **Status upstream**: GA.
- **What it gives us**: removes the `"$CLAUDE_PROJECT_DIR"` quoting dance in all 6 hook definitions in `.claude/settings.json:75,86,98,109,121,132`. Today every hook command is a string like `python3 "$CLAUDE_PROJECT_DIR"/.claude/hooks/session-start.py` — fragile if `$CLAUDE_PROJECT_DIR` ever contained whitespace, and the escaping is repeated 6×. With `args:`, each hook becomes `{ "type": "command", "args": ["python3", "${CLAUDE_PROJECT_DIR}/.claude/hooks/session-start.py"] }` — no shell parse, no quoting bugs. Also closes a hypothetical injection vector if `$CLAUDE_PROJECT_DIR` ever resolves to something hostile.
- **How to wire**: one-file change. `.claude/settings.json:69-137` — convert each `"command": "..."` to `"args": [...]` per the upstream schema. Six occurrences (SessionStart, UserPromptSubmit, PreToolUse, PreCompact, FileChanged, SubagentStop). Run a smoke test (`claude --help` then trigger each hook event) to verify the runtime accepts the exec form.
- **Replaces / consolidates with**: nothing deleted — same hooks, cleaner config form. The 6 quoted-string commands collapse to 6 args arrays.
- **Risk**: very low. The change is local to `.claude/settings.json`. Rollback = revert the file. Upstream calls this "exec form" — it's a config schema addition, not a behavior change in our hook bodies.

### ADOPT-3: PostToolUse hook + `continueOnBlock` for the DS-rule blocker

- **Upstream evidence**: Claude Code CHANGELOG 2.1.139 — *"Added hook `continueOnBlock` config option for `PostToolUse` — set to `true` to feed the hook's rejection reason back to Claude and continue the turn."* (`docs/governance/claude-updates/pending-review.md:25`).
- **Status upstream**: GA.
- **What it gives us**: today `.claude/hooks/pre-tool-use.py` (17K, `Edit|Write|MultiEdit` matcher per `.claude/settings.json:93`) inspects intended edits BEFORE they hit disk and can block them. It cannot inspect the *result* of a write nor feed structured findings back into the model after the fact. A `PostToolUse` hook with `continueOnBlock: true` would let us run a *post-write* DS audit (e.g., a focused subset of `scripts/ds-adoption-audit.py` against the just-written file) and feed structured findings into Claude's next turn rather than as a blocking error. The pre-tool-use hook stays for hard blocks (raw `<button>`, hex colors); the new post-tool-use hook covers *soft* findings ("you wrote `<Card>` with raw div children — consider Card slots") that should educate the model mid-session without aborting the edit.
- **How to wire**: two-file change.
  1. `.claude/hooks/post-tool-use.py` (new) — minimal Python that reads the PostToolUse JSON payload, extracts the just-edited file path, runs a *narrow* `scripts/ds-adoption-audit.py --file <path> --soft-rules-only` (we'd add the `--file` and `--soft-rules-only` flags to the audit script as part of this work), and on findings prints a `{"decision": "block", "reason": "<finding>"}` JSON to stdout. With `continueOnBlock: true` in the hook config, Claude reads the reason and adapts on the next turn without an outright halt.
  2. `.claude/settings.json:92-103` — add a `PostToolUse` block mirroring the existing `PreToolUse` matcher (`Edit|Write|MultiEdit`), with the new hook and `continueOnBlock: true`. The pre-tool-use.py stays for the regex rules architect promoted to BLOCK on 2026-05-11 (`architect-runs/2026-05-11-baseline.md:53-75` — datatable-no-empty-state, dialog-no-error-feedback, opacity-60-on-text-parent).
- **Replaces / consolidates with**: nothing replaced. The PostToolUse hook is purely additive — pre-tool-use.py keeps its current scope. This narrows the case for ever spawning `verification-reviewer` (`.claude/agents/verification-reviewer.md`) on every Edit, since soft findings now flow inline.
- **Risk**: medium. Risks: (a) hook noise — if every Edit prints a paragraph of findings, the model loses signal-to-noise; mitigate by capping output to top-1 finding per file. (b) audit-script perf — running the full DS audit per-edit would be slow; the `--file` flag MUST be added to constrain scope. (c) introducing `PostToolUse` requires the user-prompt-submit and pre-tool-use hook payloads to still work — Claude Code 2.1.139 supports both, but verify by running a single edit and checking both hooks fire. (d) Romit caught the `dead-link-audit.py` "always-on warn" anti-pattern in `architect-runs/2026-05-11-baseline.md:104-109` — apply the same lesson here: only soft findings with **zero current hits in the workspace** (the rules the architect already declared safe to promote) should run via PostToolUse. Anything noisier stays manual.

- **Ours**: our 6 hooks all `print(json.dumps({}))` or `print(json.dumps({"output": ...}))` to stdout only — no `sys.stderr.isatty()` checks, no curses, no raw escape sequences. `.claude/hooks/subagent-stop.py:58`, `.claude/hooks/session-start.py` (final print), etc.
- **Gap**: none — equivalent. The upstream fix protects against a class of bug we never wrote.

### "Dreaming" / sleep-time reflection (Anthropic platform announcement, May 2026)

- **Upstream**: Anthropic announced multi-agent orchestration primitives in May 2026 including a "Dreaming" capability — Managed Agents that reflect on completed work outside the active session and propose follow-ups.
- **Ours**: `.claude/agents/architect.md` (read by `python3 scripts/architect-input.py` per `CLAUDE.md` §"Architecture self-improvement"). The architect agent reads commit log + discipline log + audit hits + subagent telemetry + past architect runs and proposes structural updates, writes to `docs/governance/architect-runs/YYYY-MM-DD-<slug>.md`. The 2026-05-11 baseline (`architect-runs/2026-05-11-baseline.md`) demonstrates the loop end-to-end.
- **Gap**: upstream Dreaming runs in *Managed Agents* (a deployed-agents product surface), against *production* agents, on *Anthropic's* infra. Ours runs *in-session* (or invoked manually), against *this workspace's* governance corpus, on *Romit's* laptop. Different deployment, similar idea. Not worth swapping because the inputs and authorship are local-first by design (governance docs, discipline log, audit JSON). Revisit if Managed Agents ever exposes a hook for self-hosted corpus.

### `architect` subagent budget / completion checklist ≈ `/goal` command

- **Upstream**: Claude Code 2.1.139 — *"Added `/goal` command: set a completion condition and Claude keeps working across turns until it's met."* (`pending-review.md:20`).
- **Ours**: every subagent spec ends with an "Output checklist (verify before exiting)" block — see `.claude/agents/claude-updates-watcher.md:155-167` (this file), `.claude/agents/architect.md` (output checklist section), etc. The architect's max-3-rules + ≥2-evidence-occurrences + retire/consolidate-required rules are the same shape as a `/goal` completion contract, just enforced by spec text rather than runtime.
- **Gap**: `/goal` is *runtime-enforced* (model sees the goal as state across turns); ours is *spec-enforced* (each subagent run reads its checklist and self-verifies). Upstream has the edge for multi-turn long-form tasks but we don't currently have a multi-turn subagent (each runs one-shot). Not worth swapping until we hit a task that genuinely spans turns.
