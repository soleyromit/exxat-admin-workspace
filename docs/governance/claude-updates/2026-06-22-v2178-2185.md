## ADOPT proposals (3 / 3 budget)

### ADOPT-1: `Tool(param:value)` permission syntax — add model-gate to deny list

- **Upstream evidence**: Claude Code CHANGELOG 2.1.178 — *"Added `Tool(param:value)` syntax for permission rules to match a tool's input parameters (with `*` wildcard), e.g. `Agent(model:opus)` to block Opus subagents"* (`pending-review.md:98–99`).
- **Status upstream**: GA (main CHANGELOG body; no research-preview tag; ships in 2.1.178 stable release).
- **What it gives us**: our `.claude/settings.json` allow list currently gates subagent spawns by name — `Agent(compliance-reviewer)`, `Agent(verification-reviewer)`, etc. (`.claude/settings.json:5–17`). This requires every subagent name to be listed explicitly. The new `Tool(param:value)` syntax lets us add a *model-level* gate independently of agent name: for example, `Agent(model:claude-opus-4-8)` in the allow list would explicitly permit our chosen model, while any spawn requesting a different model (e.g., a rogue prompt that says "use claude-haiku") would require review. This closes a governance gap where model-override in a subagent prompt could silently bypass our `"model": "claude-opus-4-8"` workspace default.

  More concretely: the classifier-evaluates-subagent-spawns improvement (also CC 2.1.178) means the subagent's model parameter is now evaluated by the permission classifier *before* launch. Adding a `Tool(param:value)` rule for model pins this at the settings layer, not just at runtime.

- **How to wire**: one file, two additions to `.claude/settings.json`.
  1. In the existing `permissions.allow` array (`.claude/settings.json:3`), add:
     ```json
     "Agent(model:claude-opus-4-8)",
     "Agent(model:claude-sonnet-4-6)"
     ```
     (Allow our two approved models — opus-4-8 for main sessions, sonnet-4-6 for subagents per MEMORY.md "Design on Opus, implement-from-spec on Sonnet".)
  2. Optionally add a `permissions.deny` key (new, doesn't exist yet) with:
     ```json
     "deny": [
       "Agent(model:claude-haiku*)"
     ]
     ```
     This blocks any attempt to spawn a Haiku-based subagent, which would skip the "design on Opus" discipline.
- **Replaces / consolidates with**: does not replace any existing rule. Adds a parameter-level dimension to the existing name-based allow list. The named agent entries (`Agent(compliance-reviewer)` etc.) remain; the new `model:` rule is orthogonal.
- **Risk**: low. If an approved subagent spec changes its model field in a future edit, a session spawning it would hit the permission gate — this is the intended behavior (require review when model deviates from approved list). Rollback = remove the two added entries.

### ADOPT-2: `attribution.sessionUrl` — suppress claude.ai session link from commits/PRs

- **Upstream evidence**: Claude Code CHANGELOG 2.1.183 — *"Added `attribution.sessionUrl` setting to omit the claude.ai session link from commits and PRs in web and Remote Control sessions"* (`pending-review.md:25`).
- **Status upstream**: GA (main CHANGELOG body; 2.1.183 stable release).
- **What it gives us**: Claude Code currently appends a `claude.ai/session/<id>` link in commit messages and PR bodies created by agent sessions. For this workspace (a private Exxat engineering monorepo with shared GitHub access), the session URL leaks:
  1. The specific session identity of the Claude Code session that made the commit — linkable to the claude.ai account.
  2. A clickable link that may be confusing to non-Claude-Code reviewers (Himanshu on main-branch reviews).

  Omitting the session URL keeps commit messages cleaner and avoids accidental session-identity disclosure. The `attribution.sessionUrl: false` setting is workspace-scoped, so it does not affect Romit's personal `~/.claude/settings.json`.

- **How to wire**: one-line addition to `.claude/settings.json` at the top level (not inside `permissions` or `hooks`):
  ```json
  "attribution": {
    "sessionUrl": false
  }
  ```
  The exact JSON path is: root object → `"attribution"` key → `"sessionUrl": false`.
- **Replaces / consolidates with**: nothing deleted. Additive top-level key.
- **Risk**: zero. This is a display-only setting for commit/PR metadata. It does not affect the commit content, diff, or Claude's ability to create commits. Rollback = delete the two added lines.

### ADOPT-3: Explicit deny list for destructive git + IaC commands

- **Upstream evidence**: Claude Code CHANGELOG 2.1.183 — *"Improved auto mode safety: destructive git commands (`git reset --hard`, `git checkout -- .`, `git clean -fd`, `git stash drop`) are now blocked when you didn't ask to discard local work, `git commit --amend` is blocked when the commit wasn't made by the agent this session, and `terraform destroy`/`pulumi destroy`/`cdk destroy` are blocked unless you asked for the specific stack"* (`pending-review.md:23–24`).
- **Status upstream**: GA (main CHANGELOG body; 2.1.183 stable release). The auto-mode blocking is runtime behavior, not config.
- **What it gives us**: the upstream auto-mode blocking is heuristic — it fires when Claude judges "you didn't ask to discard local work." Heuristics can be wrong under ambiguous prompts. Our workspace has `"defaultMode": "acceptEdits"` (`.claude/settings.json:88`), which means edits are auto-accepted without per-edit confirmation. Adding explicit `Bash(git reset --hard *)`, `Bash(git stash drop *)`, and `Bash(git clean -fd *)` entries to a `permissions.deny` array makes the governance explicit and unconditional — these commands require interactive approval regardless of how confident the model is that the user asked for them.

  This directly supports the CLAUDE.md Git Safety Protocol ("NEVER run destructive git commands... unless the user explicitly requests these actions") and makes the machine-level enforcement match the written rule.

- **How to wire**: add a `permissions.deny` key to `.claude/settings.json` (currently does not exist). The `permissions` object at `.claude/settings.json:2` currently has only `allow`, `defaultMode`, and `additionalDirectories`:
  ```json
  "deny": [
    "Bash(git reset --hard *)",
    "Bash(git checkout -- *)",
    "Bash(git clean -fd *)",
    "Bash(git stash drop *)",
    "Bash(git commit --amend *)"
  ]
  ```
  Place this inside the `permissions` object alongside the existing `allow` array.
- **Replaces / consolidates with**: `.claude/agents/claude-updates-watcher.md` hard rule 7 ("NEVER run destructive git commands") is currently prose-only in the agent spec. This turns it into a machine-enforced gate at the settings layer. The prose rule remains as documentation of intent.
- **Risk**: medium-low. If Romit explicitly asks to reset, checkout, or clean, Claude will now require interactive confirmation rather than auto-accepting. This adds one confirmation step to legitimate destructive operations. It does NOT block the operations — it gates them. Rollback = remove the `deny` key or remove the specific entries. Note: `git commit --amend` is used legitimately in some workflows; if this produces friction, it can be removed from the deny list while keeping the others.
