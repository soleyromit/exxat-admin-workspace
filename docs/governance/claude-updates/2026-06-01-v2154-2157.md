## ADOPT proposals (3 / 3 budget used)

### ADOPT-1: `SessionStart` hook — add `reloadSkills: true` return key

- **Upstream evidence**: TS SDK CHANGELOG 0.3.152 — *"`SessionStart` hooks can now return `reloadSkills: true` to trigger a skill re-scan, and set the session title via `hookSpecificOutput.sessionTitle`"* (`docs/governance/claude-updates/pending-review.md:155`).
- **Status upstream**: GA (main CHANGELOG body, no "Research Preview" tag; SDK 0.3.152 mirrors Claude Code runtime support).
- **What it gives us**: two concrete improvements for one small edit.

  First, `reloadSkills: true`: our `session-start.py` hook already fires on `startup|resume|clear|compact` (`.claude/settings.json:71`). When fired on `startup` it is the right moment to force a skill re-scan — especially because our workspace skill directory (`.claude/skills/`) has 29 skills that were previously dependent on marketplace registration. Now that 2.1.157 makes `.claude/skills` auto-loaded, a fresh `startup` session that resumed after a skills edit will see the updated skill list immediately, without manual `/plugin` reload, if `reloadSkills: true` is in the hook output.

  Second, `sessionTitle`: our `session-start.py` already emits a rich `additionalContext` block (line `279`). Adding `sessionTitle` to the same output lets Claude name the session after the active product + DS profile (`"PCE · Admin DS"`, `"Exam Mgmt · Student DS"`, etc.) — useful in `claude agents` multi-session views as those mature.

- **How to wire**: one-file change.
  - `.claude/hooks/session-start.py:279` — change the final `print(json.dumps({...}))` block from:
    ```python
    print(json.dumps({
        "hookSpecificOutput": {
            "additionalContext": "\n".join(summary_lines)
        }
    }))
    ```
    to:
    ```python
    session_title = None
    if product and profile:
        session_title = f"{product} · {profile} DS"
    elif product:
        session_title = product
    output: dict = {
        "hookSpecificOutput": {
            "additionalContext": "\n".join(summary_lines)
        }
    }
    if matcher == "startup":
        output["reloadSkills"] = True
    if session_title:
        output["hookSpecificOutput"]["sessionTitle"] = session_title
    print(json.dumps(output))
    ```
  - No other file needs to change. The `reloadSkills` key is ignored by older CLI versions and harmless to add unconditionally on `startup`.
- **Replaces / consolidates with**: nothing deleted. Adds two new return keys to the existing hook output dict. The session-start hook's `additionalContext` path is unchanged.
- **Risk**: very low. `reloadSkills` is a no-op on CLI builds that don't recognize it. `sessionTitle` is display-only. Neither affects the hook's ability to emit `additionalContext`. Rollback = revert the three added lines.

### ADOPT-2: `OTEL_LOG_TOOL_DETAILS=1` env flag — wire into workspace env config

- **Upstream evidence**: Claude Code CHANGELOG 2.1.157 — *"`tool_decision` telemetry events now include `tool_parameters` (bash commands, MCP/skill names) when `OTEL_LOG_TOOL_DETAILS=1`"* (`docs/governance/claude-updates/pending-review.md:32`).
- **Status upstream**: GA (main CHANGELOG body, no "Research Preview" tag).
- **What it gives us**: our OTEL thread from the baseline run (ADOPT-1, partial) left the OTEL env block deferred ("no use case for export yet"). Since then `docs/telemetry/registry-state.json` has been actively used and `docs/telemetry/events.jsonl` is our telemetry sink. The `OTEL_LOG_TOOL_DETAILS=1` flag does not require a standing OTEL collector — it enriches the existing `tool_decision` events in the local events stream, which the `_telemetry.py` helper already writes. Concretely: once this env is set, the subagent-stop hook payload will begin to include the specific bash commands / skill names that the subagent invoked, giving the architect's input bundler (`scripts/architect-input.py`) richer per-invocation context when it reads the subagent-invocations log.

  This also closes a specific gap surfaced in `2026-05-12-baseline.md:54`: *"we'd need to know which tools each subagent called to identify the `dead-link-audit.py` anti-pattern recurrence"*. With `tool_parameters` in the `tool_decision` events, the architect can now identify subagents that call certain patterns (e.g., always-blocking hooks that never actually block) without instrumenting each hook manually.

- **How to wire**: one-file change.
  - `.claude/settings.json` — add a top-level `"env"` block (outside `"permissions"` and `"hooks"`, at the root of the JSON object):
    ```json
    "env": {
      "OTEL_LOG_TOOL_DETAILS": "1"
    }
    ```
    This is the minimal Claude Code workspace env config format. No OTLP exporter, no external collector — this flag enriches events that the existing `_telemetry.py` write path already captures.
- **Replaces / consolidates with**: nothing deleted. Augments the event payload already written to `docs/telemetry/events.jsonl` by `.claude/hooks/_telemetry.py:12`.
- **Risk**: low. The flag is additive — it only enriches event payloads; it does not change behavior or add latency. If the local CLI build does not recognize the `env` key in `settings.json`, it is silently ignored (Claude Code's settings schema ignores unknown top-level keys). Verify by checking `claude doctor` output after the edit. No production code changes.

### ADOPT-3: Concurrent hooks dispatch — add clarifying comment to session-start.py

- **Upstream evidence**: Python SDK CHANGELOG 0.2.82 (docs section) — *"Clarified that `hooks` dispatch for a given event is concurrent (all matchers fire in parallel), not sequential, preventing incorrect assumptions about ordering-dependent hooks like rate limiters gating subsequent hooks."* (`docs/governance/claude-updates/pending-review.md:258`).
- **Status upstream**: GA (docs clarification in 0.2.82, reflects runtime behavior that has been in place throughout; not a new feature — a correction to documentation of existing behavior).
- **What it gives us**: our workspace has two `PreToolUse` hooks that share the `Write|Edit|MultiEdit` matcher scope (`.claude/settings.json:92-112`):
  1. `pre-tool-use.py` (line `98`) — runs DS + WCAG hard-block rules.
  2. `exxat-brief-gate.mjs` (line `108`) — runs the Exxat brief gate.

  These are registered in the same event with overlapping matchers. The upstream docs clarification confirms that both hooks fire **concurrently**, not in the order they appear in the array. This matters because `exxat-brief-gate.mjs` currently has no awareness that `pre-tool-use.py` may simultaneously be blocking the same write — there is an implicit assumption that they run sequentially and the first block would prevent the second from doing redundant work. In practice with concurrent dispatch, both hooks will independently decide to block or pass, and Claude receives both verdicts. This is not a bug, but it is an undocumented assumption in our hook architecture.

  The fix is a two-line comment in each hook header (and a note in `.claude/settings.json`) so future contributors understand the dispatch model and don't write ordering-dependent logic (e.g., "don't check X if hook Y already blocked" — that pattern cannot work).

- **How to wire**: two comment-only edits.
  1. `.claude/hooks/pre-tool-use.py` — add at the top of the module docstring (after line 1's `"""PreToolUse hook`):
     ```
     Dispatch note: PreToolUse hooks with overlapping matchers fire CONCURRENTLY
     (per Agent SDK Python 0.2.82 docs). Do not assume this hook runs before or
     after exxat-brief-gate.mjs. Both fire in parallel; Claude receives both verdicts.
     ```
  2. `.claude/hooks/exxat-brief-gate.mjs` — add an equivalent comment at the top.
  3. `.claude/settings.json:92` — add a comment line (JSON doesn't support inline comments natively, but the settings.json is already using pure JSON; this could instead be captured as a separate `docs/governance/hook-dispatch-model.md` note if Romit prefers). Recommendation: skip the settings.json edit; the hook file headers are sufficient.
- **Replaces / consolidates with**: nothing deleted. Comment-only change. Prevents a future hook-ordering assumption bug rather than fixing an existing runtime issue.
- **Risk**: zero runtime risk. Comment-only edits. If Romit prefers a single governance note over editing two hook files, the same information can live in `docs/governance/hook-dispatch-model.md` (new, 5 lines) — that is an equally valid form.

Before applying, verify with `claude doctor` that the key is accepted without warning. 3. **ADOPT-3 format choice**: two hook file comment edits vs. a new `docs/governance/hook-dispatch-model.md`. The hook-file approach is more discoverable (future contributors see it at the point of authoring a new hook); the governance doc is more centralized. I recommend the hook-file approach, but Romit may prefer to keep all governance notes in `docs/governance/`.
4. **PostToolUse soft-rules**: the baseline ADOPT-3 was deferred for "≥2 weeks of subagent-invocations data." That window has passed. Is `docs/governance/subagent-invocations.log` accumulating rows? If yes, the next watcher run should revisit this as a candidate ADOPT.
