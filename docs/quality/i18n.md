## Locale strategy

Phase 1 = `en-US` only. Phase 2 candidates based on customer geography:
- `en-GB`, `en-CA`, `en-AU` (English variants ÔÇö may need only minor terminology differences)
- `es-MX`, `es-ES` (Spanish ÔÇö clinical Spanish materials common in PT/PA programs)
- `fr-CA` (Canadian French ÔÇö required for Quebec accreditation contexts)

When a Phase 2 locale ships, file an ADR:
- Why this locale
- Glossary translation owner
- QA path for the new locale

## Architecture (Phase 1 ÔÇö ready, not active)

### Message catalog format

Per app, messages live under `apps/<product>/<app-type>/messages/<locale>.json`:

```json
{
  "common": {
    "save": "Save",
    "cancel": "Cancel",
    "loading": "LoadingÔÇª"
  },
  "course": {
    "list_title": "My Courses",
    "empty_state": "You haven't been assigned to a course yet.",
    "add_button": "Add course"
  }
}
```

Namespacing: `common`, then per-feature (`course`, `assessment`, `student`, etc.).

Match the per-product glossary at `apps/<product>/docs/content.md` ÔÇö the same terms appear here with the same definitions.

### Locale resolution

Locale resolves at the school-configuration level (per workspace ADR-002 LMS-first thinking):

1. School config has `locale: 'en-US'` (or future locale code)
2. Each app reads school config on session start
3. Faculty/student see the school's chosen locale; no per-user override Phase 1

When LMS provides locale data, use that. When manual, the admin sets it during first-run.

### Library choice

Phase 1 ships catalog stubs. When activating a second locale:

| Option | Trade-offs |
|---|---|
| `next-intl` | Best Next.js App Router integration; SSG-friendly |
| `react-intl` | Battle-tested, ICU MessageFormat support |
| `i18next` | Most flexible; ecosystem-heavy |

Decision deferred to ADR when Phase 2 locale ships.

### RTL preparation (I18N-003)

CSS uses **logical properties** (start/end) instead of physical (left/right) so RTL languages would just work:

| Don't | Do |
|---|---|
| `padding-left: 8px` | `padding-inline-start: 8px` (or Tailwind `ps-2`) |
| `margin-right: 16px` | `margin-inline-end: 16px` (or `me-4`) |
| `text-align: left` | `text-align: start` |
| `border-left: 1px solid` | `border-inline-start: 1px solid` |

Tailwind's logical-property classes (`ps-*`, `pe-*`, `ms-*`, `me-*`, `start-*`, `end-*`) work today.

Phase 1 doesn't need actual RTL testing, but new code should use logical properties from the start.

## Rules (DESIGN.md ┬º4 I18N)

| ID | Rule | Gate |
|---|---|---|
| I18N-001 | All user-facing strings live in message catalogs (no hardcoded strings in components) | Code review |
| I18N-002 | Locale resolution happens at school-configuration level; no per-user override Phase 1 | Architecture review |
| I18N-003 | CSS uses logical properties (start/end) for inline directions; not physical (left/right) | Code review + lint |
| I18N-004 | New locale activation requires ADR (locale, owner, QA path) | Governance |

## Phase 1 light deliverable

For Phase 1, each app gets a stub `messages/en-US.json` with the most-used strings. No locale switcher UI yet. The architecture is ready for Phase 2 activation.

(Don't create empty stubs for not-yet-scaffolded products ÔÇö wait until product code starts.)

## Anti-patterns

- ÔØî Hardcoded English strings in components (Phase 1 might tolerate; Phase 2 breaks)
- ÔØî Physical-direction CSS (`left: 0`, `padding-left: 8px`) ÔÇö pre-emptively use logical
- ÔØî Date/time formatting via `toLocaleString()` without locale arg ÔÇö pass the resolved locale
- ÔØî Currency formatting via string concatenation ÔÇö use `Intl.NumberFormat` with locale
- ÔØî Hardcoded English in error messages or empty states (per CONTENT-002, CONTENT-003) ÔÇö these go through the catalog too
