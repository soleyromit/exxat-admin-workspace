## User Flows

- **Download window flow:** Assessment published by coordinator → coordinator sets download window (e.g., opens 1 week before exam, closes 2 hours before) → students log in during that window → system downloads encrypted exam to their device → on exam day, students enter classroom, receive password, unlock exam → take exam offline if needed → submit when reconnected
  > "When an assessment is published by a course coordinator and is made available on the student's view, the coordinator's setup defines the download window. The download window stops before they enter the lecture hall. If they have not done it, they are in trouble. The expectation is that the exam is already on their local device before they come."

- **CTA for student:** Download action must be EXPLICIT with a clear call-to-action. Not automatic/silent.
  > "All those nuances we have to think. If you are telling me we will not support the download, but students will log in one hour before the exam, they will forget about it. So we have to give them an explicit way to say there is an action needed from them."

- **Copy from previous assessment:** Course coordinator copies prior year's assessment → sections + questions are copied → coordinator can move questions between sections → regenerates faculty assignments for sections (may differ from prior year)
  > "I think a way for us to copy the structure in terms of sections and the questions, but giving them the ability to edit, reorganize questions, move them to other sections, would help."
  > "The sections is what you will need to copy. But which section gets assigned to which faculty — all of that may change."

## Design Decisions

- **Offline download is MANDATORY for Phase 1 / January launch:**
  > "This was always part of ExamSoft parity when we reviewed everything. This concern did come until now where when was this decision taken that we will not support download? [...] We think that download should be provided."

- **Download = encrypted file in browser/device storage, not a separate app.** Different from ExamSoft's separate downloaded app, but achieves same outcome.
  > "It's not a separate software which they have to download. The download actually happens in the browser itself."

- **Offline architecture:** Students log in → system caches exam encrypted in local storage → password unlocks on exam day → submit when reconnected
  > "They log in, they get a chance to download. We are authenticating them and they're able to download the exam. Now they take that exam when they get the password. Even if they are offline they are in lockdown browser. When it's time to submit — they need a connection. If they're not able to submit because of lack of connectivity, whenever they're connected again is when it gets submitted."

- **Accessibility — Phase 1 requirements:**
  - 200% magnification (path to 400% in future)
  - Text-to-speech (check ExamSoft parity first)
  - Dyslexic-friendly font support
  - High contrast color combinations
  > "200% magnification, speech to text, dyslexic font support, high contrast color ratios, high contrast color combinations. Text size up to 200-400%. These are the few things that we want to give in the first phase."

- **Accommodations in exam setup:** Course coordinator sees which students have accommodations for that course, what accommodations they have, and plans accordingly.

- **ExamSoft parity table is mandatory:** Must create explicit table: ExamSoft features we match / features we don't support (with rationale) / our differentiators beyond ExamSoft.
  > "We need to do this exercise for the entire exam management module, everything that we are. We need to be very clear on what where do we stand with respect to ExamSoft. And whatever decisions we take, have to be very conscious once we know this is parity, but we have decided not to do — that needs alignment from Aarti."

- **Differentiators must be explicitly called out** for launch/marketing purposes.
  > "Whatever we are providing extra, we should consciously make all of our team members aware. So we should know that equally important — the differentiator. Because now when we create launch plan, we will use the differentiators to market it."

## Scope Constraints

- **IN (Phase 1):** Offline download with configurable window; accessibility (200% magnification, dyslexic font, high contrast); copy-from-previous with section structure
- **IN (Phase 1):** Assessment setup options with parity table vs ExamSoft
- **DEFERRED (Phase 2):** 400% magnification
- **DEFERRED (Phase 2):** Full lockdown browser
  > "Phase 1: offline download + accessibility. Phase 2: lockdown browser."
- **DEFERRED (Phase 3+):** Live proctoring, monitored exams

---

## Data / Entity Rules

- **Download window:** coordinator-configured; two dates: window_open, window_close (relative to exam time)
- **Exam encryption:** exam is encrypted at download; password unlocks it at exam time
- **Assessment setup options:** each is a configurable field (time limit, password, download window, randomization, accommodation time multipliers)
