## User Flows

- **Curricular assessment loop (the four-question model):**
  1. What must I teach? (standards/accreditation requirements → curriculum design)
  2. What am I teaching? (course objectives → courses → content coverage)
  3. How am I assessing? (questions → assessments → what's actually tested)
  4. What are students learning? (scores → feedback → loop back to curriculum)
  > "What must I teach? Then what am I teaching? Mhmm. So what am I teaching? Then here you have your questions, and you group these questions into assessments. Then the question is, what am I assessing? And ideally I should be assessing them on what I'm teaching. If I don't assess them, I don't have evidence that students learned."

- **Course page in EM:** Faculty clicks a course → lands on assessment list → secondary tab shows course detail (objectives, content, reading material, etc.). NOT a blank form — everything already set up.
  > "When they come to or they talk about exam management module, they are coming there with the sole purpose of making questions, creating assessments, administering those, getting reports."

## Design Decisions

- **Three roles in Phase 1:** Admin, faculty, student. No eight admin variants.
  > "I just want clarity amongst everybody on the project that we are at the beginning only going to worry about three roles and three stakeholders."

- **Two faculty sub-roles:** Course coordinator (can create/edit assessments, manage all sections, publish) and instructor (can edit their assigned sections only). Collaborator = third option (read-only or limited).
  > "There is a course coordinator and then there is an instructor. At least two will be in phase one."
  > "So if we need control over — can you make an assessment? Can you administer an assessment? — we need control over those."

- **Base entities that MUST exist in EM (all listed):** Courses, terms, course offerings, students, faculty, accommodations, content areas, competencies, standards, assessment types, roles.
  > "Master list of courses, terms, course offerings, students, faculty, registrations, accommodations, content areas, competencies, standards, faculty list, roles list, types of assessments."

- **Assessment types must be defined for Phase 1:** What types do we support (quiz, midterm, final, take-home, pop quiz)? Which types need specific configurations? Lockdown browser → Phase 2. Proctored → Phase 3.
  > "Take home exams may have something different. Quizzes may have something different. Proctored exams can have something different. We have to make a list of what type of assessments we are going to support in phase one."

- **Course page landing = assessments, not course info:** When in EM and clicking a course, first land on assessments tab. Course details (objectives, readings, content areas) are secondary tab but reachable in one click.
  > "Select course 594. My landing page can talk about my assessments. And then there can be a tab about course details or measures or all of that. But landing has to be on assessments."

- **Course objectives NOT mandatory in EM:** If they don't populate objectives, AI will have limited capability, but system doesn't block them.
  > "We cannot make it mandatory that the objectives have to be filled in. [...] Without doing the mapping, AI will have very arbitrary results. But the user has that flexibility."

- **QB is linked to base course, not offerings:** Confirmed again here.
  > "The question bank shells are going to be different [from the main course]. And the main course with actual students registered is going to be different."

- **AI gap analysis depends on LMS integration:** For content-based question creation (where AI reads course materials), LMS integration is prerequisite.
  > "LMS integration is a prerequisite for AI to offer content-based question creation."

- **Student data in EM = exam performance only:** Student performance in EM = course/assessment scores, competency strengths/weaknesses based on EM data. Not full 360/GPA.

## Scope Constraints

- **IN (Phase 1):** All base entity pages (courses, terms, offerings, students, faculty, accommodations, content areas, competencies); three roles; two faculty sub-roles
- **IN (Phase 1):** Course page landing on assessments; one-click access to course details
- **IN (Phase 1):** Assessment types defined and configured per type
- **DEFERRED:** All 8+ admin role variants — just three in Phase 1
- **DEFERRED:** Curriculum mapping inside EM — use Prism/existing curriculum mapping module
  > "Don't create curriculum mapping here. It's so much confusion."

## Data / Entity Rules

- **Base entity hierarchy:** Program → Term → Course → Course Offering → Assessment → Section → Question (in QB, linked to section)
- **Roles:** admin (program-level), course_coordinator (course-level, full access), instructor (course-level, section-limited), student
- **Content areas, competencies, standards:** Program-level entities; mapped to course objectives; used as labels on questions
